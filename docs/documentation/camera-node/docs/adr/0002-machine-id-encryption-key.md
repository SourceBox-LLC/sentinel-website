# ADR 0002: Machine-ID-derived key for at-rest encryption

- **Status:** Accepted
- **Date:** 2026-04
- **Deciders:** Camera Node maintainers

## Context

Camera Node has two kinds of secret material on disk in `data/node.db`:

1. **The cloud API key** — a `nak_*` token that authenticates the node to Command Center. Compromise = an attacker can push fake camera data into the customer's org.
2. **Recording + snapshot blobs** — the actual pixels, encrypted starting in v0.1.16 (commit `8f8f549`). Compromise = the customer's footage is exposed.

Both need to be encrypted at rest. A reasonable read of the node.db file (a stolen disk, a cloned VM, a misplaced SD card from a Pi) must produce useless ciphertext, not plaintext.

We considered four options for where the encryption key comes from:

1. **User passphrase.** Operator types a passphrase on first run; we derive the key with PBKDF2/scrypt/argon2. Strong threat model, works against any attacker including ones with code-execution on the host. Cost: every Camera Node boot now needs a human typing a passphrase, which kills the "install and walk away" install flow and breaks unattended Docker / systemd deployments.
2. **OS keychain** (Keychain on macOS, DPAPI on Windows, libsecret/Secret Service on Linux). Battle-tested, threat model identical to whatever the OS gives you. Cost: three platform integrations to write and maintain, plus headless-Linux deployments where there's no D-Bus session and the keychain effectively doesn't exist.
3. **Hostname-derived.** SHA-256(hostname + app_salt). What v1 of Camera Node shipped. Cheap to derive, no user interaction. Cost: hostnames change easily — DHCP lease changes, OS reinstall, Docker container rename. A hostname change silently invalidates the key and bricks the existing DB.
4. **Machine-ID-derived.** SHA-256(OS-managed machine identifier + app_salt). The OS-managed identifier is `/etc/machine-id` on Linux, the `MachineGuid` registry key on Windows, `IOPlatformUUID` on macOS. All three are persistent across reboots, hostname changes, and software updates; all three change exactly when the OS is reinstalled or the disk is moved to a new machine.

## Decision

**The encryption key is `SHA-256(machine_id_bytes || application_salt)`.** Deterministic per host, regenerated on demand, never written to disk.

Implementation lives in `src/storage/database.rs::derive_key`. Cached per-process via `OnceLock` so repeated calls don't re-read the OS source. Fallback chain on Linux: `/etc/machine-id` → `/var/lib/dbus/machine-id` → fail loudly.

Both the API key (hex-encoded ciphertext, via `encrypt_value`) and the recording/snapshot blobs (binary, via `encrypt_bytes`) use the same derived key. Domain separation is implicit in the format prefixes (hex vs the `OSE\x02\x02` magic — the v0.1.18+ blob-format V2; the earlier V1 prefix `OSE\x02\x01` is still readable) — nothing in either path can decrypt a payload the other path wrote.

### v1 → v2 migration

Camera Node v0.1.x originally used hostname-derived keys (v1). When we moved to machine-id (v2) in commit `4f...` (March 2026), we kept a one-time fallback path: `decrypt_value` tries the v2 key first, falls back to the v1 hostname-derived key, and if v1 succeeds, the caller re-encrypts the value with v2 on the next save. Subsequent boots never touch the v1 path. The migration is silent from the operator's perspective.

The fallback path is `derive_key_legacy()` in the same module. It's still called by `decrypt_value` (string config) but **not** by `decrypt_bytes` (BLOB encryption shipped in v2-only).

## Consequences

**Positive:**
- Stolen DB on a different machine is unreadable. The attacker needs both the file and code execution on the original host to re-derive the key.
- No user interaction. The setup wizard never prompts for a passphrase, the systemd unit boots cold, the Docker container starts fine without `-e PASSPHRASE=...`.
- No third-party crypto integrations to maintain. `aes-gcm` from the `RustCrypto` org and the OS-supplied machine identifier are the entire dependency surface.
- Survives every routine operational event: reboots, hostname changes, OS upgrades, package upgrades, DHCP changes, Docker container restarts.
- Fails loud and fast if the source identifier is missing — `derive_key()` returns `Err` and the app refuses to start, rather than silently producing a different key and corrupting future writes.

**Negative:**
- **OS reinstall destroys access to the existing DB.** The machine ID is regenerated by the OS installer; the key changes; the existing ciphertext becomes unreadable. We document this in `setup/` and the README. The mitigation is "back up your recordings if you care about them after reformatting."
- **Cloning a VM clones the key.** If a customer takes a snapshot of a running Camera Node VM and spins up two copies, both copies can read the same DB. By design — the key follows the OS image — but worth knowing if your threat model includes "attacker exfiltrates a VM snapshot."
- **No revocation story.** If the machine-id leaks (somehow — they're not normally secret), there's no "rotate the key" command. We'd ship a `rotate-encryption-key` subcommand if this ever became a real problem, decrypting the existing DB with the old key and re-encrypting with a new one. Not built yet because there's no demand.
- **Linux Docker containers** without a host-bind-mounted `/etc/machine-id` get a fresh ID on every container creation, which means the DB doesn't survive `docker rm`. The Docker docs in the repo's `README.md` note the bind-mount requirement.

**Neutral:**
- The `application_salt` is a hardcoded constant in the source, not a per-install secret. It's not load-bearing for security — its only job is to keep the derived key from being identical to other software that might happen to hash the same machine ID. An attacker who reads our source can compute the same intermediate; the security still rests entirely on knowing the machine ID, which they don't.
- The chosen primitive is AES-256-GCM rather than ChaCha20-Poly1305 because Pi 4 has hardware AES (ARMv8 crypto extensions) and we want hardware acceleration where it's available. Software AES on Pi Zero 2W (no ARMv8) is still well within budget for our blob sizes — we benchmarked at ~70 MB/s on Pi 4 hardware-accelerated and ~25 MB/s on Pi Zero 2W software, both well above the ~1.5 MB/s recording bandwidth.

## Revisiting this decision

Re-evaluate if any of the following become true:

- We need **portable backups** — a customer wants to back up their Camera Node's DB and restore it on a different machine. The current design says "no, that's a feature." If "yes, that's a feature" wins, we add a passphrase-based key-wrapping option (the data-encryption key is still per-machine, but it's wrapped in a passphrase-derived KEK that travels with the backup).
- A customer is genuinely worried about "attacker steals running VM snapshot." Then we layer OS-level disk encryption (LUKS / BitLocker / FileVault) underneath the application-level encryption. We document this as defense-in-depth in `docs/runbooks/` already; if it becomes mandatory, we'd ship a setup-wizard check that refuses to proceed without OS-level encryption.
- We start running Camera Node in environments where machine identifiers aren't persistent (some serverless platforms, specific virtualization configs). Then we'd need either a passphrase fallback or an opt-in stored key file, with appropriate warnings about the security implications.
- A new Pi platform ships without a stable machine-ID path (e.g. an immutable-OS Pi where `/etc/machine-id` is regenerated on every boot). We'd need a per-disk fallback identifier.
